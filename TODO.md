## Bugs

- [x] **[MEDIUM]** Add first-run spotlight coachmark tour with seeded sample project (desktop)
  - Description: First-time users currently land on an empty Void screen ("NO PROJECTS YET" + a single "Create your first project" button) and have to discover the rest of the app on their own — projects, due-date pills, expand chevron, Pomodoro, music, settings, theme toggle, sidebar. Replace that cold start with a dismissible multi-step coachmark tour that runs the first time the app loads with no saved data. To guarantee every step has a real DOM target to anchor to, silently seed a "Getting started" sample project (a real project the user can rename, edit, or delete like any other) with 3–4 starter todos before the first tour step renders. Each tour step dims the rest of the UI with a full-viewport overlay, cuts out a single highlighted element, and anchors a small purple-bordered callout next to it (`STEP N OF M`, a short heading, one or two sentences of guidance, Skip / pagination dots / Next ›). The callout must be positioned against the live `getBoundingClientRect` of its target with collision-aware placement (flip to opposite side or recenter if it would clip the viewport) — the prior mockup positioned the Step 1 callout against an element that didn't exist in the empty state, which is the failure mode this gating prevents. Proposed steps: (1) the seeded sample project entry in the sidebar, (2) one of the seeded todo rows' due-date pill, (3) the expand-description chevron on a seeded row, (4) the sidebar "+" project button, (5) the Pomodoro toggle in the navbar, (6) the music toggle (`#musicToggle`) in the navbar, (7) the settings toggle (`#settingsToggle`) in the navbar. Advance on Next or when the user interacts with the highlighted element; the final step ends with a "You're set" closer that leaves the sample project in place for the user to keep or delete. Persist a `todoapp_onboardingComplete` flag in `localStorage` so it never auto-runs again; expose a "Replay welcome tour" entry in the sidebar settings panel for re-triggering (replay does not re-seed the sample project — only the first run does). Constraints: no new dependencies (build the overlay + cutout + callout in vanilla JS — a fixed full-viewport `<div>` with four mask rects around the target's bounding rect), close on Escape and on backdrop click outside the callout, recompute target geometry on `resize` and `scroll`, and keep the overlay above all existing popovers and modals. The seeding logic belongs in `listLogic.js` (gated on a separate `todoapp_sampleSeeded` flag so a user who deletes the sample project doesn't get it back); the tour controller and DOM lives in `main.js` / `style.css`; `index.js` triggers the seed-then-tour sequence on first boot. Mobile is intentionally out of scope for this entry — a follow-up will adapt the flow for narrow viewports.
  - File: `toDoList_main/src/main.js`, `toDoList_main/src/listLogic.js`, `toDoList_main/src/index.js`, `toDoList_main/src/style.css`, `toDoList_main/tests/listLogic.test.js`
