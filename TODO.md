- [x] **[MEDIUM]** Add JSON export and import buttons that hit Supabase directly
  - Description: Replace the cut Drive sync feature with a simpler, more user-empowering backup mechanism: two buttons in the settings menu that export the user's entire dataset to a downloadable JSON file, and re-import a JSON file to replace their Supabase data. This gives users a real escape hatch — they can leave the app whenever they want with their data in a portable format they own — and serves as a manual backup affordance for users who want belt-and-suspenders against data loss. Unlike Drive sync, this isn't a continuous background sync; it's an explicit, one-click action the user takes when they want a snapshot. The export reads directly from Supabase (not localStorage) to ensure the file reflects authoritative server state. The import writes directly to Supabase, then triggers a re-hydration so the UI reflects the imported state.
  - Architecture: (1) Export reads from Supabase via two `select * from projects` and `select * from todos` queries scoped to the user via RLS (no explicit user_id filter needed — RLS handles it). The result is serialized as JSON with shape `{ version: 1, exportedAt: <iso>, projects: [...], todos: [...] }` and triggered as a browser download via a synthetic anchor element with `href=URL.createObjectURL(blob)`. (2) Import accepts a JSON file via a hidden `<input type="file">`, parses the contents, validates the shape, shows a confirmation modal warning that this will replace all current data, and on confirm runs a sequence of Supabase calls: truncate-all (delete user's projects, which cascades to todos via FK), then bulk-insert the imported projects, then bulk-insert the imported todos. After the import completes, re-hydrate by calling the existing `hydrateFromSupabase` (which now operates on the freshly-imported data) and dispatches `listLogicHydrated` so the UI rebuilds.
  - UI placement: Add a new "Data" section to the settings menu (replacing the slot the cut Drive section occupied) with two rows: "Export to JSON" and "Import from JSON". On the mobile Settings modal, add the same two rows as a new Data section. Each row uses the existing `createDrawerActionRow` (mobile) or `buildSettingsMenuItem` (desktop) helpers so the visual treatment matches the surrounding rows. Export row triggers an immediate download with filename `todolist-export-<YYYY-MM-DD>.json`. Import row opens the hidden file input, then on file selection shows a confirmation modal: "Replace all your data with the contents of this file? Your current X projects and Y todos will be deleted and replaced." with Cancel / Replace buttons. Replace fires the actual import.
  - Filename: `todolist-export-2026-05-24.json` for an export on May 24 2026. If multiple exports happen the same day, the browser handles uniqueness via download numbering (`-1`, `-2`, etc.) — no custom timestamping needed.
  - JSON shape: `{ "version": 1, "exportedAt": "2026-05-24T12:34:56.789Z", "projects": [{ "id": "uuid", "name": "Project 1", "color": null, "position": 0 }, ...], "todos": [{ "id": "uuid", "project_id": "uuid", "title": "Item 1", "description": "...", "due_date": "2026-05-31", "priority": "2", "position": 0, "completed": false, "recurrence": null }, ...] }`. The `version: 1` field is for future format migrations. On import, validate that `version === 1`; reject any other value with a user-facing error "Unsupported export format". Other validation: `projects` and `todos` must be arrays, each project must have `id` and `name`, each todo must have `id`, `project_id`, and `title`. Reject malformed files with "Invalid export file — could not parse" and surface the error in the confirmation-replaced area.
  - Behavior - export: User clicks Export to JSON in the settings menu. The menu closes. A loading state isn't needed for the typical case (queries complete in <500ms). On success, the browser triggers a download of `todolist-export-<date>.json`. On failure (Supabase error, network drop), show a toast: "Export failed — try again". User can retry without consequence.
  - Behavior - import: User clicks Import from JSON in the settings menu. The menu closes. A native file picker opens (filtered to `.json` files). User selects a file. The file is read and parsed. If parsing fails, a toast shows "Invalid export file — could not parse" and the flow ends. If parsing succeeds, a confirmation modal shows with the project/todo counts from the file and the warning copy above. Cancel dismisses the modal. Replace triggers the import: delete all user's projects in Supabase (todos cascade via FK ON DELETE CASCADE), then bulk-insert imported projects, then bulk-insert imported todos. On any step failure, show a toast "Import failed — your data was not changed" — but only if the failure happened before deletion; if deletion succeeded but inserts failed, the user is in a bad state and needs a more aggressive error: "Import failed — your data may be in an inconsistent state. Try importing again or contact support." After all inserts complete, call `hydrateFromSupabase` to refresh the UI.
  - Implementation: Create a new module `toDoList_main/src/jsonImportExport.js` with three exports: `exportToJson()` (returns a Promise resolving to nothing; triggers the download as a side effect), `importFromJson(file)` (returns a Promise resolving to `{ projectCount, todoCount }` or rejecting with an error), and `validateImportShape(json)` (synchronous, returns `{ valid: true }` or `{ valid: false, error: '...' }`). Wire the export row to call `exportToJson()` on click. Wire the import row to open a file input on click, then on file change call `validateImportShape` synchronously and either show the confirmation modal (on valid) or the toast (on invalid). The confirmation modal's Replace handler calls `importFromJson(file)`, awaits it, and on success calls `hydrateFromSupabase()`. Reuse the existing `showConfirmModal` helper from modals.js for the confirmation modal. Reuse the existing `showDriveToast`-equivalent for toasts — but Drive is being cut in the sibling entry, so first refactor `showDriveToast` into a generic `showToast` in a shared module (or inline a minimal toast helper in jsonImportExport.js).
  - Edge cases: (1) Empty export (user has no projects) — the file still downloads, just with empty arrays. Not an error. (2) Import of an export from another user — the IDs in the file are UUIDs from that user's account. When inserted into the current user's account via RLS-enforced INSERT, the `user_id` column on `projects` is auto-set to the current user (per the schema's RLS policy default). The IDs from the file are preserved as-is, which is fine since UUIDs are globally unique. (3) Import where the user already has data — the truncate-and-replace flow handles this correctly per the confirmation modal's warning.
