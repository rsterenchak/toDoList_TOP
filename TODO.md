## Features

- [x] **[HIGH]** Refactor listLogic.js to route persistence through Supabase (Phase 5 — backend migration)
  - Description: Pivot `listLogic.js` from "localStorage is the source of truth" to "Supabase is the source of truth, localStorage is the boot cache, in-memory `allProjects` is the render mirror." Architecture decisions locked in Phase 1 (optimistic write + realtime push, last-write-wins on `updated_at`) and Phase 5 planning (hybrid sync API + async hydrate, localStorage as boot cache + Supabase reconcile, filter blank placeholders at the persistence boundary, completed recurrence clones as full `todos` rows). No schema changes, no UI changes in `main.js`, no signature changes to the existing read API. **In-memory shape gains an `id` field per project and per todo:** each project entry becomes `{ id: uuid, items: [{ id: uuid, tit, desc, due, pri, pos, completed, recurrence }, ...], color: null|string }`. The existing API still keys by name in the rendering layer; internal persistence resolves to UUIDs. **Boot sequence (in `index.js`):** (1) `listLogic` IIFE runs as today and exposes the synchronous API immediately with `allProjects` populated from localStorage and sanitized (existing migration + sanitization passes are retained verbatim — this is the offline cache); (2) after the Phase 4 boot gate confirms a session exists, `index.js` awaits a new `listLogic.hydrateFromSupabase()` call that fetches all the user's projects and todos, runs the **reconciliation pass** (project exists in Supabase but not local → add; exists locally but not Supabase → push to Supabase; exists in both → `updated_at` comparison picks winner), rewrites `allProjects` in place, persists the merged result to localStorage, and dispatches a new `listLogicHydrated` CustomEvent that `main.js` listens for to do a one-shot full re-render; (3) `index.js` then subscribes to Supabase realtime via `listLogic.subscribeToRealtime()` so subsequent remote changes flow in. Hydration is gated on session — sign-out cancels any in-flight hydrate and clears `allProjects`. **Mutation funnel (`persistMutation` helper, all 14 mutation functions route through it):** every user-mutation function (`addProject`, `removeProject`, `editProject`, `addToDo`, `removeToDo`, `removeToDoByItem`, `insertToDoAt`, `reorderProject`, `reorderToDo`, `sortCompletedToBottom` when it actually rewrites, `setProjectColor`, `setRecurrence`, `advanceRecurringTodo`, `seedSampleProject`, `seedSampleTodos`) becomes: (a) mutate `allProjects` in memory exactly as today, (b) call `saveToStorage()` exactly as today (localStorage write + existing event dispatch retained for the offline cache), (c) call `persistMutation({ op, table, payload })` which fires the appropriate Supabase call — INSERT for new rows, UPDATE for existing, DELETE for removals, with per-operation translation from in-memory shape to row shape, (d) on Supabase response, if it was an INSERT, assign the server-returned UUID onto the in-memory object's `id` field so subsequent updates target the right row; if it was a failure, log via `console.warn` and do not roll back (Phase 6 adds the offline queue + visible sync-issues indicator). **Blank-placeholder filtering** happens inside `persistMutation` and `hydrateFromSupabase`: any item with `tit === ''` is skipped on write, and blanks are re-injected at index 0 of every project's items array after hydrate via `sortCompletedInPlace`. **UUID assignment** uses `crypto.randomUUID()` at the moment of in-memory creation (browsers since 2021 support this); Supabase's `gen_random_uuid()` default never fires because we always pass an explicit `id`. **Date format conversion:** `due` ("M-D-YYYY" string) ↔ `due_date` (ISO date) translation lives in the persistence boundary inside `persistMutation` and `hydrateFromSupabase` — the in-memory shape keeps the "M-D-YYYY" string format the existing render code expects. **Recurring task completed clones (`advanceRecurringTodo`)** fire two separate writes through `persistMutation`: one INSERT for the completed clone (a brand new row with `completed: true`, `recurrence: null`, due locked to the date that was satisfied) and one UPDATE for the original (new `due_date`). Both clones and originals are full `todos` rows with their own UUIDs; the recurring task's history is N rows in the database sharing `title` and `project_id`. **`replaceAllProjects` (used by the existing JSON import flow)** becomes a `truncate + bulk-insert` against Supabase under the hood: DELETE all current user's projects (cascades to todos), then INSERT the new tree. Migration in Phase 6 will reuse this path. **`saveToStorage` semantics are preserved** so the existing `driveSyncStateChanged` event dispatch (renamed inline to `dataChanged` for clarity, with a backward-compat alias for any external listener) still fires after every mutation — this is what the offline cache and the (now-vestigial) Drive sync indicator both listen for. The Drive listeners stay wired but no-op since Drive auto-sync is commented out per the earlier dev-friction fix and slated for full removal in the post-Phase-5 cleanup pass. **Realtime subscription (`subscribeToRealtime`):** uses two `supabase.channel('public:projects').on('postgres_changes', { event: '*', schema: 'public', table: 'projects' }, handler)` subscriptions, one each for `projects` and `todos`, both filtered server-side by `user_id` and `project_id IN (...)` via RLS so we don't have to filter client-side. Incoming INSERT events check if the row's `id` is already in `allProjects` (self-echo from optimistic write) — if yes, ignore; if no, add it. UPDATE events overwrite the in-memory row's mutable fields. DELETE events splice the row out. All three dispatch `dataChanged` so the UI re-renders. Subscription lifecycle: opened on hydrate, closed on sign-out, reopened on sign-in. **Sign-out hook:** `listLogic.handleSignOut()` clears `allProjects`, clears localStorage's `allProjects` key, cancels realtime subscriptions, cancels any in-flight `persistMutation` promises. Called from the `onAuthStateChange` listener Phase 4 wired in `index.js`. **Tests:** new `tests/listLogicSupabase.test.js` source-level pins (1) `persistMutation` is called from every mutation function with the correct `op` + `table` arguments, (2) blank-titled items are filtered on write and re-injected on hydrate, (3) `hydrateFromSupabase` reconciliation picks the row with the newer `updated_at`, (4) `advanceRecurringTodo` fires two separate writes, (5) `subscribeToRealtime` opens two channels and ignores self-echo events by `id`, (6) `handleSignOut` clears state and cancels subscriptions. Existing tests in `tests/listLogic.test.js` continue to pass — the public API contract is unchanged, only the persistence layer underneath. **Out of scope for this entry:** the offline queue (Phase 6 — `persistMutation` failures currently `console.warn` without retry), the localStorage→Supabase migration on first login (Phase 6 — reuses `replaceAllProjects`), removal of the now-dead Drive code (separate cleanup pass after Phase 5 lands), conflict-surface UI for diverged writes (still last-write-wins for now). **Implementation note for the automation:** `main.js` exceeds 25k tokens; navigate with grep + offset/limit pagination. The mutation funnel pattern is consistent across all 14 functions — write `persistMutation` once and apply it mechanically; resist the urge to special-case unless tests force it.
