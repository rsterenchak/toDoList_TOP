# TODO LIST

- [ ] **[HIGH]** Switcher question count repaints in an unbounded MutationObserver loop
  - Type: bug
  - Description: `updateAllProjectQuestionCounts` in `main.js` writes `badge.textContent = String(count)` on every pass. The `textContent` setter replaces the badge's text node unconditionally (no diff), so it emits a childList mutation record on a node inside `sideMain`'s subtree. The `questionCountObserver` registered alongside it observes `sideMain` with `childList: true, subtree: true`, so that write re-triggers the very callback that produced it: observer → paint → mutation → observer. MutationObserver callbacks run as microtasks, so the queue never drains and the tab hangs. The in-code comment reasons only about the `class` toggle — correct, since `class` is not in the `attributeFilter` — and misses that `childList` is newly observed here. The existing `.projBadge` writer does the same unconditional `textContent` write and is safe only because `footObserver` observes `sideMain` with attributes and no `childList`. The loop is gated on `count > 0`: with no waiting questions no badge is created, the `if (badge)` guard skips the write, and the paint settles — so the app looks healthy until the first `needs_words` row lands, which is precisely the case the feature exists for. Every revert control lives inside the app, so the hang is not recoverable in-app.
  - Behavior: A project with waiting questions paints its amber count once and settles. Repaints triggered by a realtime push, a row rebuild, or a project rename each run a bounded number of passes and stop. No project count changes visually. The count remains live and the switcher keeps rendering its full project list under every failure path the current try/catch already covers.
  - Implementation notes:
    - Guard the text write on inequality so a steady-state pass emits no mutation: compute `const next = String(count);` and write only when `badge.textContent !== next`. This is the whole fix — the loop terminates because a settled badge stops producing records.
    - Do NOT guard `aria-label`, `title`, or the `classList` toggles. `attributeFilter` is `['value']`, so none of those are observed and none contribute to the loop; adding guards there is noise.
    - Leave the badge-creation path as-is — it is already guarded by `if (!badge)` and only fires on the first paint for a given row, so its childList record costs exactly one extra pass rather than an unbounded run.
    - Do not narrow the observer's `childList`/`subtree` options as the fix. `subtree` applies to both childList and attributes in a single `observe()` call and cannot be split, and dropping `subtree` would break the rename repaint that reads `#projInput`'s `value`. The equality guard is the correct seam.
    - Regression coverage goes in the existing `switcher wiring — source (main.js)` describe block in `tests/projectSwitcherQuestionCount.test.js`, as a source assertion that the text write is inequality-guarded. `main.js` is closure-bound and not instantiable in jsdom (per CLAUDE.md), so a behavioural test of the loop is not available — the source pin is the same strategy the rest of that block already uses.
  - Out of scope: The count's semantics (`needs_words` only), `getWaitingQuestionCounts`, the all-projects cache and its fetch, the realtime wiring, the `.projQuestionCount` styling and grid variants, and the existing `.projBadge` writer — which is unaffected because `footObserver` does not observe `childList`.
