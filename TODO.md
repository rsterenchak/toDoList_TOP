- [x] **[MEDIUM]** claudeSheet: attach a task to the chat so the whole conversation stays scoped to it — Completed: 2026-07-22
  - Type: feature
  - Description: `openChatWithSeed` pre-fills the composer with a task's title, description, and pending question, but that context is one-shot — it lives in the first message and is gone by the second, so a multi-turn conversation about a task loses track of which task it is. Add a durable attachment instead: a task can be attached to the chat session, renders as a scope chip above the composer, and rides along with every turn until it is detached or the session is reset. This is the difference between seeding a conversation and scoping one, and it is what makes chat usable for working a single task rather than only opening one.
  - Behavior: A committed task row offers a "Discuss" action that opens the Claude sheet with that task attached. The sheet shows a scope chip above the composer carrying the task title and a detach control; with nothing attached the chip reads as unscoped so the state is never ambiguous. While a task is attached, every message sent — Fast or Deep — carries the task's title and description as context, so follow-up turns need no re-explanation. Detaching clears the chip and stops the context riding along; the conversation itself is untouched. "New Chat" clears the attachment along with the transcript. The attachment survives closing and reopening the sheet within a session, so switching to Tasks to look at something and coming back does not silently drop scope. Attaching a different task replaces the current one — at most one task is attached at a time.
  - Implementation notes:
    - The attached task must ride in the TURN-VARIABLE tail of the request, not the repo-stable block. The prompt-cache layout is four ordered blocks (system prompt with 1hr TTL → repo-stable context → iterate seed → attachments and turn-variable tail); attach and detach are frequent, so placing the task above the stable context invalidates the cached prefix on every scope change. Grep how file and image attachments are assembled into the request body and put the task in that same tail position.
    - Reuse the existing attachment chip pattern rather than inventing a second one — file and image attachments already render chips near the composer with a remove control. Grep that markup and mirror it, so a scoped task reads as a sibling of the other attachment types and inherits the same layout behavior.
    - Persist the attachment the way the sheet already persists its own state — grep the `todoapp_`-prefixed key the run records use (`RUNS_KEY`) and follow that shape. Store the todo id, not the item object; resolve the title through listLogic at render time so a renamed task does not display a stale chip. If the id no longer resolves (task deleted, project switched away), clear the attachment and fall back to unscoped rather than rendering a dangling chip.
    - Do not write to `agent_queue` from this path. Attaching is a client-side scope, not a pipeline action, and must not create a queue row or interact with triage. Generate is the path that creates queue rows; these two stay independent.
    - `claudeSheet.js` already tracks `activeChatRepo` and `activeHandoffRow` as module-level session state — add the attachment alongside them and clear it in the same place "New Chat" clears the transcript, so the reset paths can not drift apart.
    - The row-side "Discuss" action belongs beside the existing per-row affordances. `claudeSheet.js` is imported by `agentView.js` already, but `toDoRow.js` importing it directly risks the `toDoRow → claudeSheet → modals → toDoRow` cycle that `inject.js` documents and deliberately avoids — reach it through a registered handler the way `setViewerCardTapHandler` does, or verify the cycle does not form before adding the import.
    - `style.css`: the scope chip reuses the existing attachment-chip tokens and the 10px radius convention; no new color tokens. This entry touches `style.css`, so it runs with no other style-touching entry in flight.
  - Out of scope: Emitting entries from chat back onto the attached task — the natural follow-up, deliberately separate. `openChatWithSeed` and the Agent board's "Discuss in chat" hand-off, which keep seeding text and are not converted to attachment in this entry. Auto-attaching based on what is selected or open. Attaching more than one task, or attaching projects, runs, or files-as-tasks. The Runs tab, the model strip, voice dictation, and the file/image attachment paths themselves. Any change to what the model is asked to do with the context — this entry only carries it.

- [ ] **[MEDIUM]** Task row: surface triage's pending question and answer it without leaving the list
  - Type: feature
  - Description: When triage needs input it parks its `agent_queue` row in `needs_words` with a question, and the only place to see or answer it is the Agent board. That is the last state genuinely blocked on the user that lives on another surface — every other part of the loop (phase, generate, viewer, chat scope) now reaches the task row. Surface the pending question on the originating task's row and let it be answered there, routing through the same write path and the same auto-fired triage sweep the board already uses.
  - Behavior: A task whose linked `agent_queue` row is in `needs_words` renders an `⌁ ASKING` badge in amber, and its description panel shows triage's question above an answer field with a Send action. Sending writes the answer through listLogic and fires the triage sweep exactly as the board does, after which the badge clears and the row returns to its normal phase display. Unsent answer text survives a row rebuild (sort, filter, project switch) rather than being silently dropped. Rows with no linked queue row, or whose row is in any other state, are unchanged. The Agent board keeps rendering and answering `needs_words` exactly as it does today — this adds a second place to answer, it does not remove the first.

    - Extend `derivePhase` with an `asking` phase rather than adding a second state channel to the row. Extract the queue-row cache and its subscription into a small module both can import, rather than reaching across into the view.
