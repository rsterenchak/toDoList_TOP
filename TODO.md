- [ ] **[HIGH]** Investigate and fix service worker update-available notification not surfacing on mobile (and possibly desktop)
  - Description: The PWA's service worker update-available notification flow does not appear to be reliably surfacing new builds to users, leaving installed clients running stale code indefinitely. Discovered when investigating a "Drive sync doesn't work on mobile" report and finding mobile was running a pre-consolidation build (still showed separate Export/Import to Drive menu rows instead of the single Sync button shipped weeks ago, and exhibited the Connect-discards-token bug already fixed in a later PR). The user had no way of knowing mobile was stale until they noticed the UI mismatch against desktop. This is a serious silent-failure: every shipped fix is invisible on a client until the user manually hard-refreshes or unregisters the service worker via DevTools — neither of which a normal user knows to do. Existing infrastructure context: `index.js` registers the service worker and "wires the update-available notification" per its own bootstrap comment, so the intended flow exists in code somewhere. The mechanism should be: new build deployed to GitHub Pages → Workbox-managed `sw.js` detects new assets at the next page load → fires `updatefound` on the existing registration → new SW reaches `installed` state in the background → app surfaces a user-visible prompt ("New version available — refresh to update") → user taps refresh → `skipWaiting` activates the new SW → page reloads into the new build. Investigation phase (not yet a fix — this entry is scoped to diagnose and document before deciding the actual remediation): (1) read the current SW registration code in `index.js` and the Workbox config in `sw.js` to determine whether `updatefound` listeners are wired and what they do; (2) determine whether the update-available DOM prompt actually renders — search `main.js` and any UI modules for the notification element by ID or class and confirm its render path; (3) test the full update cycle in a controlled environment: deploy a build with a known version bump, open the app on a previously-installed client, observe whether the prompt appears within 1-2 page loads; (4) repeat on iOS Safari specifically — iOS PWA service worker handling has documented quirks (background sync deprioritized, `skipWaiting` sometimes ignored, cache-first strategies can pin old builds); (5) check the deploy workflow (`.github/workflows/deploy.yml` per repo conventions) for cache-busting headers or asset-hash-in-filename setup — if the bundled JS filenames don't include content hashes, the SW may not detect that anything changed; (6) review the Workbox runtime caching strategy — `CacheFirst` for HTML or the main JS bundle would lock users into the cached version permanently; the entry-point HTML must use `NetworkFirst` or `StaleWhileRevalidate` to let the SW pick up new builds. Hypotheses to confirm or rule out, in order of likelihood: **(a) the update prompt is wired but never gets shown** because `updatefound` fires but the listener's callback either silently fails, is gated on a condition that never holds (e.g., `if (registration.waiting)` evaluating false in some code paths), or renders into a DOM container that doesn't exist on certain viewports — would explain why desktop sees updates but mobile doesn't, if the prompt's host element only exists in the desktop layout; **(b) the SW is caching the HTML aggressively** with `CacheFirst`, so the new SW's existence is never even discovered because the browser keeps serving the old `index.html` that references the old asset bundles; **(c) bundle filenames lack content hashes** so the SW thinks nothing changed across deploys; **(d) iOS-specific lifecycle bug** — the SW updates correctly but iOS won't activate the new version without a full app close/reopen, and there's no visible cue telling the user to do that. Each hypothesis has a different fix shape; the investigation determines which one applies. Possible remediation directions (NOT prescriptive yet — pick based on what investigation finds): if (a), audit the notification render path and ensure the host element exists across all layouts; if (b), change the HTML cache strategy to `NetworkFirst` and verify SW activation; if (c), update Webpack config to emit content-hashed filenames for the entry chunks (note: `CLAUDE.md` says "no changes to webpack.config.js unless the task explicitly requires it" — this would qualify); if (d), add a manual "Check for updates" entry in the ghost menu's HELP section that calls `registration.update()` and surfaces the result, giving users an escape hatch even when the automatic path fails. A user-visible "running version X" indicator in the footer or About section would also help users self-diagnose: if they see "running version 1.0.8" on mobile and "1.1.3" on desktop, they know to hard-refresh. Acceptance criteria (post-investigation, may be refined): (1) deploying a new build with a version bump causes any open client to surface a user-visible update prompt within at most 2 page loads (not requiring a hard refresh or DevTools intervention); (2) the prompt works identically on desktop Chrome, mobile Safari (iOS), and mobile Chrome (Android); (3) the prompt offers a clear "Refresh now" action that activates the new SW and reloads the page; (4) a visible "version X" string somewhere in the app's UI (footer, About modal, ghost menu HELP section) lets a user confirm what version they're running without opening DevTools; (5) the deploy workflow emits content-hashed asset filenames (if the investigation finds this is missing); (6) the README documents the update flow so future contributors know the contract. Tests: in a new `serviceWorkerUpdate.test.js`, pin (a) the registration code subscribes to `updatefound`; (b) the callback wires `statechange` on the new worker; (c) when the new worker reaches `installed` AND there's an existing controller (i.e., this is an update not a first install), the user-visible prompt renders; (d) clicking the prompt's refresh button calls `skipWaiting()` on the new worker and reloads the page. Out of scope for this entry: refactoring to a different SW library (e.g., switching off Workbox); offline-mode improvements unrelated to update discovery; per-asset cache invalidation strategies beyond what Workbox provides out of the box. Changelog: add a 1.1 entry once the actual remediation lands — wording depends on what was wrong, but along the lines of "Fixed: the app now reliably prompts you to refresh when a new version is available, instead of silently running the cached version forever."

Re-prompt OAuth on each session — do not persist the access token to `localStorage`.
