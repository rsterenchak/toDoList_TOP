- [x] **[HIGH]** Fix duplicate-key 409 when editing existing todo title and pressing Enter
  - Description: Editing an existing committed todo's title and pressing Enter fires a Supabase POST `/rest/v1/todos` that returns 409 with PostgreSQL error code 23505 ("duplicate key value violates unique constraint todos_pkey"). The expected behavior is that editing a title should fire UPDATE (PATCH), not INSERT (POST) — the row already exists in Supabase and only its `title` column should change. Console diagnostic during Phase 5 verification confirms: `[persistMutation] called: insert todos {...}` with the existing todo's UUID, immediately followed by `[persistMutation] insert error: { code: '23505', message: 'duplicate key value violates unique constraint "todos_pkey"' }`. The 409 response means the row was correctly NOT created (Supabase enforced the unique constraint), but the user's edit silently fails to persist to the backend. localStorage still updates via the keyup handler's `listLogic.saveToStorage()` call, so the edit appears to work until the next reload from Supabase wipes it. Root cause is unknown — the toDoInput keydown handler in `toDoList_main/src/toDoRow.js` runs the full commit ceremony (mutate `item.tit`, set `item.pri = 2`, delete `dataset.originalBlank`, call `listLogic.saveToStorage()`, then either `appendNewToDoRow` or `focusBlankToDoInput`) regardless of whether the row is being committed for the first time or being re-edited. Some path inside this ceremony — most likely `appendNewToDoRow` → `sortCompletedToBottom`'s blank-placeholder recreation, OR a stale code path in `addToDo` that re-INSERTs an already-existing row — is firing `persistMutation` with `op: 'insert'` instead of `op: 'update'` when it should be a no-op (the row already exists and isn't changing) or an UPDATE.
  - Diagnostic step before implementing: Add `console.trace('[persistMutation] stack');` immediately after the existing `console.log('[persistMutation] called:', ...)` line in `toDoList_main/src/listLogic.js`. Reproduce the bug by editing an existing committed todo's title and pressing Enter. The browser console will show the call stack identifying which function in `listLogic.js` fired the INSERT — that function is the bug location. Remove the `console.trace` line after diagnosis is complete and the fix is in. Suspect functions (ranked by likelihood): `sortCompletedToBottom` (may re-INSERT a blank placeholder whose UUID already exists in Supabase), `addToDo` (may be called from a path that should be calling `editToDo` or equivalent), or some position-sync helper that confuses INSERT with UPDATE on title-edit reorders.
  - Behavior: Edit an existing committed todo's title (click input, change text, press Enter or blur). Network shows PATCH `/rest/v1/todos?id={uuid}` returning 204, NOT POST returning 409. Console shows `[persistMutation] called: update todos {...}` with the existing UUID, NOT insert. The Supabase `todos` table's `title` column for that row reflects the new value after the request completes. Subsequent refreshes display the edited title. No 23505 errors fire in the console for the title-edit flow.
  - Implementation: Determined by stack-trace diagnosis. Likely fixes (one or more of these may apply): (1) If `sortCompletedToBottom` is the offender, audit its blank-placeholder recreation logic to ensure it uses the existing blank's UUID from the in-memory array rather than generating a fresh one that may collide with Supabase; if it must INSERT a fresh blank, it must first delete the old one. (2) If `addToDo` is being called incorrectly from the title-edit path, refactor `toDoRow.js`'s Enter keydown handler to distinguish "first commit of a blank placeholder" from "edit of an existing committed row" and route to different code paths — first-commit goes through `addToDo`/`commitBlankPlaceholder`, edits go through `editToDo` or just `saveToStorage` + a direct UPDATE through `persistMutation`. (3) If the bug is in a position-sync helper that calls INSERT instead of UPDATE for already-existing rows, change `op: 'insert'` to `op: 'update'` and verify the payload shape matches the update branch's expectations in `persistMutation`. Whichever fix applies, add a regression test in `tests/listLogicSupabase.test.js` that edits an existing item's title and asserts `persistMutation` was invoked with `op: 'update'`, not `op: 'insert'`.
  - Acceptance criteria: (1) Editing an existing committed todo's title fires exactly one PATCH `/rest/v1/todos?id={uuid}` request returning 204, no POST and no 409; (2) The updated title persists across page refresh and remains in the Supabase `todos` table when checked via Supabase Table Editor; (3) No `[persistMutation] insert error: { code: '23505' }` warnings fire on the title-edit flow; (4) First-commit of a brand-new todo (typing into blank placeholder and pressing Enter on a project with no other todos) still fires POST `/rest/v1/todos` returning 201 — no regression of the working first-commit case; (5) All other Phase 5 mutation paths (add second/third todo, check completion, delete, project add/rename/delete) continue to fire their expected requests with no regression; (6) Regression test added.
