- [x] **[MEDIUM]** Add dedicated "Connect to Drive" entry point and silent re-auth on app load
  - Description: Currently Drive auto-sync only arms after the user manually clicks Export or Import once in a session — the OAuth grant is a side effect of a data-transfer action. This conflates "set up sync" with "perform an action," and means users who just want sync running in the background have to fake an export first. Add a dedicated entry point that establishes the OAuth grant without performing any data transfer, plus a silent re-auth attempt on app load so returning users in good standing skip the click entirely. Entry-point placement: a new row in the ghost menu's DRIVE section, positioned ABOVE the existing Export and Import rows. Label depends on connection state: "Connect to Drive" when not yet signed in this session, "Signed in — auto-sync on" (non-clickable, dimmed) when signed in and auto-sync is armed, "Reconnect to Drive" when a prior session's silent refresh failed or the token expired mid-session. Click handler calls `getAccessToken()` directly via `driveAuth.js` — no upload, no download, no `replaceAllProjects`, no `saveToStorage`, no `lastLocalMutationAt` write. On success: arm auto-sync (`_autoSyncArmed = true`), trigger an immediate sync attempt against current state (so the indicator reflects truth right away), and re-render the menu row to the "Signed in" state. On failure (user dismissed popup, network error): show the existing Drive toast, leave auto-sync disarmed, leave the menu row labeled "Connect to Drive" so the user can try again. Silent re-auth on app load: in the existing app-load Drive-state-check path (the one-shot query that runs at bootstrap), call `getAccessToken({ silent: true })` BEFORE the Drive list query. The `{ silent: true }` opts hint asks Google Identity Services to attempt token issuance without showing a popup — Google's GIS library supports this via `prompt: 'none'` on the token client. If silent refresh succeeds, the cached token is established, the Drive query proceeds, auto-sync arms automatically, and the user sees a green indicator immediately on load with zero clicks. If silent refresh fails (no prior grant on this browser, grant expired, user signed out of Google), fail silently — no toast, no popup, no console error — and leave auto-sync dormant with the "Connect to Drive" row visible. This means a returning user in good standing has zero-click sync; a first-time user or someone after a long absence sees the connect prompt and clicks once. Implementation notes: `driveAuth.js`'s `getAccessToken` gains an opts parameter accepting `{ silent: true }`; under that flag it constructs the GIS token client with `prompt: 'none'`. Without the flag, behavior is unchanged (interactive popup as today). The app-load path uses the silent variant; the "Connect to Drive" menu click uses the interactive variant. The auto-sync arming flag from the auto-sync entry (`_autoSyncArmed`) is set to true by either a successful silent refresh OR a successful explicit Connect click — so the loop doesn't care which path armed it. Edge case to handle: if silent refresh succeeds but the subsequent Drive query fails (network blip, API down), surface a failed state but don't disarm — the next mutation tick will retry and likely succeed. Failed silent refresh ≠ failed sync; we should only disarm on actual sync failures, not auth failures during the silent path. Acceptance criteria: (1) on a fresh install with no prior OAuth grant, opening the app shows a green-or-amber indicator only after the user clicks the new "Connect to Drive" row once; (2) on a returning user with a valid prior grant, opening the app shows green within a few hundred ms of load with zero clicks; (3) on a returning user whose grant has expired, opening the app shows the "Connect to Drive" row again, silently, with no console errors; (4) clicking "Connect to Drive" never triggers an export, import, or any mutation of local data — it's auth-only; (5) the menu row label updates reactively: "Connect to Drive" → (click) → "Signed in — auto-sync on" without a menu reopen; (6) Export and Import rows continue to work standalone for users who want to force a direction, and clicking them still arms auto-sync as before (so behavior is additive, not a regression). Tests: in a new `driveAuth.test.js` block (or extend existing), pin that `getAccessToken({ silent: true })` constructs the GIS token client with `prompt: 'none'`; pin that `getAccessToken()` with no opts uses the default interactive prompt. In `driveAutoSync.test.js`, pin that a successful silent refresh on app load arms `_autoSyncArmed`; pin that a failed silent refresh does NOT arm and does NOT throw or toast. In a new `connectToDriveMenuRow.test.js` (or extend `driveExport.test.js`'s settings-menu block), pin: row exists above the Export row in the DRIVE section; clicking it calls `getAccessToken()` (interactive variant) and NOT `exportTodosToDrive` or `importTodosFromDrive`; label reactively updates after success; row is hidden when signed in OR shows "Signed in — auto-sync on" as a non-clickable status row (whichever matches the implementation choice — pin the actual behavior). Changelog: add a 1.1 entry "New: Drive sync now connects automatically on app load for returning users. First-time users have a dedicated 'Connect to Drive' option in the menu — no more needing to export or import to get started."

- [x] **[MEDIUM]** Add "Export to Google Drive" option to ghost menu — Completed: 2026-05-22
  - Description: Add a new "Export to Drive" menu item directly below the existing "Export JSON" row in the ghost popover menu. Clicking it triggers a Google OAuth 2.0 flow (via Google Identity Services, using PKCE) prompting the user to sign in and authorize the app with the non-sensitive `drive.file` scope, then uploads the same JSON payload that the existing Export JSON produces to the user's Drive root via a multipart POST to `https://www.googleapis.com/upload/drive/v3/files`. On success, show a toast matching the existing toast styling with an "Open in Drive" link to the uploaded file's `webViewLink`. Re-prompt OAuth on each session — do not persist the access token to `localStorage`. Reuse the existing export filename pattern (`todoapp-export-YYYY-MM-DD.json`) and the same JSON serialization path so both export methods produce byte-identical files.
  - Behavior:
    1. User opens ghost popover menu, sees "Export to Drive" as a new row below "Export JSON".
    2. First click in a session opens Google sign-in popup; subsequent clicks in the same session reuse the in-memory token until it expires (~1 hour), at which point re-auth runs silently if possible, otherwise re-prompts.
    3. While the upload is in flight, the menu row shows a subtle loading state (text dim + spinner) and is disabled to prevent double-uploads.
    4. On success: toast "Exported to Drive" with "Open in Drive" link (opens `webViewLink` in a new tab).
    5. On failure (denied consent, network error, API error): toast with a clear error message; no partial state left behind.
  - Implementation notes:
    - Requires a Google Cloud project with an OAuth 2.0 Client ID configured for the GitHub Pages origin. Document the setup steps in a comment block at the top of the new driveExport module so a fresh fork knows what to provision.
    - Use the Google Identity Services library loaded lazily on first click via a `<script>` tag injection (mirrors the lazy-load pattern from the music player's YouTube IFrame API), not bundled with Webpack — avoids a hard dependency on a Google CDN at build time.
    - Use the `drive.file` scope only (not `drive`) so the app stays in the non-sensitive scope tier and avoids Google's security review process.
    - Extract the JSON serialization logic from the existing Export JSON handler into a shared helper so both code paths produce identical output.
    - Build the OAuth client ID into a module-level constant; do not hardcode in `main.js`.
  - Out of scope: folder picker (always dumps to Drive root), token persistence across sessions, two-way sync, import-from-Drive (separate future entry), accounts/multi-user.
