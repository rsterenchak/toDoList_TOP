- [ ] **[HIGH]** Add visibilitychange, focus, and 60s visible-tab poll triggers to Drive auto-sync
  - Description: Currently Device B only re-queries Drive when (a) the ghost menu opens, (b) a local mutation fires the 10s debounce, or (c) the boot-time silent re-auth fires. This leaves an unbounded staleness window where B reads `synced` while A's edits sit on Drive, and worse, B's next mutation races A's push and surfaces as `diverged` even though the user wasn't trying to fight A — they just weren't informed. Add three new triggers that all funnel through the existing `performAutoSync()` orchestrator: (1) `visibilitychange` listener that fires `performAutoSync()` when `document.visibilityState` transitions to `'visible'`, (2) `window` `focus` event listener as a desktop-tab-switch backstop (some browsers fire focus but not visibilitychange), (3) a `setInterval` poll at 60s cadence that fires `performAutoSync()` only while the tab is visible. All three triggers are gated by `getCachedAccessToken()` being non-null so no OAuth popup risk; with no cached token, they're silent no-ops. The existing `_inFlight` guard in `performAutoSync()` coalesces near-simultaneous triggers (visibilitychange + focus on desktop tab return). The existing decision tree handles every outcome: synced→noop, behind→silent pull, ahead→push, diverged→indicator flips and conflict popover available via Sync row. No new UI, no toast, no new state — the badge already communicates state changes. Pause the poll when hidden by gating the `setInterval` callback on `document.visibilityState === 'visible'` rather than tearing down and recreating the interval. Tests cover: visibilitychange→performAutoSync wiring, focus→performAutoSync wiring, 60s interval registered with visibility gate, hidden tab skips the poll body, no-token short-circuit, and the in-flight coalescing case.
