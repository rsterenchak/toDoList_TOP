- [x] **[MEDIUM]** Surface a generated draft that has not been looked at — Completed: 2026-07-23
  - Type: feature
  - Description: Generate dispatches to Actions and takes roughly a minute, so the user closes the task and does something else. When the derived entry lands in the description, nothing announces it — the task renders identically to an idea that was never touched, and the draft is found only by reopening the task on a hunch. REVIEW and ASKING both reach the user through the badge word on mobile rows; a landed draft is the third state genuinely blocked on a human and is the only one still silent. Give it the same treatment: a `⌁ DRAFTED` badge state, cleared once the task has been opened. The `draft_seen_at timestamptz` column already exists on `todos` — treat it as present and wire to it exactly as `entry_reviewed_at` is wired; do not gate on verifying it, since the repo cannot see the live schema and a previous run stalled indefinitely trying.
  - Behavior: A task whose linked `agent_queue` row is in `drafted` and whose todo has no `draftSeenAt` renders `⌁ DRAFTED` in amber, on mobile as well as desktop, joining REVIEW and ASKING as a blocked-on-you state. Opening the task's description editor clears it — the signal means "you have not looked at this yet", and looking is the whole of what it asks for, so no separate dismiss control. Tapping the badge opens the description editor, matching what ASKING already does. The state is per-task and does not clear for other generated tasks. Once cleared it does not return for that generation; a subsequent Generate on the same task sets it again.

Ranking is `asking` > `drafted` > the four marker-derived phases. Do NOT add it to `PHASE_RAIL_ORDER` or `PHASE_RAIL_LABELS` — like `asking`, it is a triage-queue fact rather than a pipeline node, and the rail resolves it to its underlying stage rather than growing a fifth node.

- [x] **[MEDIUM]** Project switcher: show which other projects have agent work waiting — Completed: 2026-07-23
  - Type: feature
  - Description: The three blocked-on-you states (REVIEW, ASKING, DRAFTED) all render on task rows, which means they are only visible in the project currently on screen. With seven projects, a triage question parked in one and a generated draft landed in another are both silent until the user happens to switch to them — and the whole point of dispatching work asynchronously is that the user leaves. Surface a per-project count of waiting agent work in the project switcher, so the signal reaches the user without visiting each project in turn. Deliberately covers only the two `agent_queue`-derived states; see Out of scope for why REVIEW is excluded and how the label reflects that.
  - Behavior: Each project in the switcher shows an amber count when it has waiting agent work — the number of its tasks whose linked `agent_queue` row is in `needs_words`, plus those in `drafted` whose todo has no `draftSeenAt`. A project with none shows nothing (no zero badge). The currently-open project shows its own count the same way, so the switcher reads consistently rather than special-casing the current one. The count is live: a realtime `agent_queue` push updates it without a re-render or a project switch. It is labelled to name agent work specifically, not "needs you", because shipped-but-unreviewed entries are not included and a general label would under-report. Tapping a project switches to it exactly as it does today; the count adds no new tap behavior.
  - Implementation notes:
    - FIRST grep the project switcher's render path — the sidebar builds `.selectedProject` rows containing `#projInput`, and `agentQueueStore.js`'s `resolveSelectedProjectName` reads that structure. Mount the count in whatever builds those rows; do not invent a second project list.
    - `agentQueueStore.js` owns the fetch. Add an all-projects load alongside the existing per-project one: `fetchQueueRows` filters on `project_id`, but RLS already scopes `agent_queue` to the user, so the same select with no `.eq` returns every project's rows in one round trip. Cache it SEPARATELY from `_rows` — that cache is the render path's synchronous source and re-scoping it to all projects would change what `getQueueRowForTodo` returns.
    - Follow the existing stale-guard and stub-client patterns in `fetchQueueRows` exactly: wrap the chain in try/catch and resolve to `[]` on error, since the test client's `.select()` has no `.eq` and resolves immediately. A failed load must degrade to no counts, never a throw.
    - The DRAFTED half needs each todo's `draftSeenAt`, which means the todos for OTHER projects must be in memory. Verify this before relying on it — grep how `listItems` is populated and whether hydrate loads every project or only the selected one. If other projects' todos are NOT in memory, count only `needs_words` and say so in the PR body rather than fetching todos per project.
    - Reuse the store's existing realtime channel and `onQueueChange` listener rather than opening a second subscription — the channel is already app-lifetime and fires on every `agent_queue` change regardless of project. Have the push refresh the all-projects cache alongside the per-project one, from one fetch where possible.
    - Do not import `agentView.js` or `toDoRow.js` from the switcher; read the store directly, the way the row layer does.
    - `style.css`: the count reuses the existing `#ffbd5e` amber and the established radius conventions; no new tokens. This entry touches `style.css`, so it runs with no other style-touching entry in flight.
  - Out of scope: Counting REVIEW (shipped-but-unacknowledged) cross-project. That state resolves through the per-repo TODO.md marker cache, so covering seven projects would mean seven GitHub reads on load — a real cost, and a separate decision. The label must therefore name agent work rather than implying it covers everything blocked on the user. Also out: any notification outside the app, a global aggregate badge on the chat button, auto-switching to a project with waiting work, the task rows' own badges, and the Agent board.
  - File: `toDoList_main/src/agentQueueStore.js`, `toDoList_main/src/main.js`, `toDoList_main/src/style.css`
