- [ ] **[HIGH]** Fix Drive sync indicator stuck on amber because import path bumps lastLocalMutationAt
  - Description: The Drive sync indicator never returns to green after a successful import (and likely after export too). Root cause: when Drive import runs, the success path calls into `listLogic.js`'s replace-all flow which mutates the data model and triggers `saveToStorage`, which unconditionally writes `lastLocalMutationAt = Date.now()`. That write lands a few milliseconds *after* `lastDriveSyncedAt` was set from the Drive file's `modifiedTime`, so the comparison `lastLocalMutationAt > lastDriveSyncedAt` reads true and the indicator computes `ahead` instead of `synced`. Confirmed in a live session: `lastDriveSyncedAt = 2026-05-23T10:14:28.783Z`, `lastLocalMutationAt = 2026-05-23T10:14:35.839Z` — the mutation stamp sits 7s after the sync stamp despite no user edits between them. Fix by making the `lastLocalMutationAt` write suppressible from sync-initiated callers. In `listLogic.js`, give `saveToStorage` an optional `{ fromSync: true }` opts argument; when the flag is set, the function persists the projects payload as usual but skips the `writeLastLocalMutationAt` call. Thread the flag through every mutation entrypoint that the Drive import pipeline touches — at minimum `replaceAllProjects` (or whichever helper `importTodosFromString` ultimately calls to swap in the imported data); add the same `opts` parameter so callers from the Drive flow can opt out of the mutation-timestamp bump. Update `driveImport.js`'s success branch to pass `{ fromSync: true }` through the replace call. Also audit `driveExport.js` for the same pattern — if the export orchestrator calls anything in `listLogic.js` that hits `saveToStorage` (e.g. a pre-upload snapshot path), thread the flag there too. Beyond the flag, reorder the writes in `driveImport.js` so `writeLastDriveSyncedAt(file.modifiedTime)` runs *before* the data replace dispatches the `driveSyncStateChanged` recompute event, so the live recompute never reads a state where the data has been swapped in but the sync marker hasn't caught up. Acceptance criteria: (1) after a successful import, `lastLocalMutationAt <= lastDriveSyncedAt`, and the indicator computes `synced`; (2) after a successful export, same invariant holds; (3) a genuine user mutation immediately after sync still bumps `lastLocalMutationAt` and flips the indicator to `ahead` — i.e., the flag only suppresses the write for sync-initiated `saveToStorage` calls, never for user-initiated ones. Tests: in `listLogic.test.js`, pin that `saveToStorage()` with no opts writes `lastLocalMutationAt` (existing behavior unchanged) and that `saveToStorage({ fromSync: true })` does NOT write it; pin that `replaceAllProjects(..., { fromSync: true })` forwards the flag and leaves `lastLocalMutationAt` untouched while still persisting projects. In `driveImport.test.js`, pin that the post-import success branch passes `{ fromSync: true }` through and that the post-import state computes as `synced`, not `ahead`. In `driveExport.test.js`, add a parallel pin if the export orchestrator was found to hit `saveToStorage` during the audit. Add a changelog 1.1 entry "Fixed: Drive sync indicator now correctly returns to green after a successful import or export."
