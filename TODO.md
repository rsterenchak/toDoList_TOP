  - Description: The Drive sync indicator still computes `ahead` immediately after a successful import, despite the previous PRs writing `lastDriveSyncedAt` correctly and threading `{ fromSync: true }` through `replaceAllProjects`. - [ ] **[LOW]** Add Drive auto-sync first slice (app load + debounced mutation, dormant until first click, diverged auto-pauses)
  - Description: Replace manual push/pull as the default Drive sync interaction with an auto-sync layer that resolves direction based on the existing three-timestamp comparison (`lastDriveSyncedAt`, `lastLocalMutationAt`, `driveModifiedTime`). The user opts in implicitly by clicking the Drive sync icon once in a session — that click runs the existing manual flow AND establishes the in-memory OAuth token. From that moment forward auto-sync is "armed" for the session, and the system handles push/pull silently. Triggers (first slice — keep narrow): (1) on app load, query Drive once IF a cached token exists from a prior session; otherwise wait dormant until the user's first manual click of the session arms the loop; (2) on every local mutation, schedule a debounced 10-second timer; on timer fire, run a sync attempt. Skip window-focus and periodic-poll triggers for v1 — revisit if real usage shows they're needed. Decision logic on a sync attempt: compute `localAhead = lastLocalMutationAt > lastDriveSyncedAt` (free, localStorage only) and `driveAhead = driveModifiedTime > lastDriveSyncedAt` (requires a Drive list query; cache the result from the most recent app-load or sync-completion to avoid one query per mutation). Branches: `!localAhead && !driveAhead` → noop (already synced); `localAhead && !driveAhead` → auto-push (call the existing `exportTodosToDrive` orchestrator); `!localAhead && driveAhead` → auto-pull (call the existing `importTodosFromDrive` orchestrator, skipping its confirmation modal — auto-sync runs without a confirm step since the user opted in by enabling auto-sync); `localAhead && driveAhead` → diverged → **auto-pause and never silently overwrite**. The diverged state disarms the auto-sync loop until the user manually resolves it via the Drive popover, which gains two new buttons "Push to Drive (overwrite Drive copy)" and "Pull from Drive (overwrite local)" with copy that explicitly states what gets discarded. Indicator semantics extend the existing three-state model with a fourth state: synced (green `ti-cloud-check`), behind/ahead — for the brief window between mutation and the next debounced sync attempt — (amber `ti-cloud-up`, same as today), diverged (red `ti-cloud-x` + tooltip "Drive and this device both changed — open the menu to resolve"), and failed (red `ti-cloud-off` + tooltip naming the failure: "Auto-sync failed: network error" / "Auto-sync failed: sign-in expired"). Failure handling: any thrown error inside the debounced sync attempt sets state to `failed` and disarms the auto-sync loop for the session (don't retry-storm a broken Drive); the user re-arms by clicking the Drive icon. Token expiration (~1hr) is treated as a failure — surface "Sign-in expired" and let the next manual click silently refresh. Manual popover repurposed: when the indicator is `synced`, the popover shows a read-only "Last synced N minutes ago" plus a "Sync now" button that runs an immediate sync attempt (bypasses debounce). When the indicator is `diverged`, the popover shows the two overwrite-warning buttons described above. When `failed`, the popover shows the failure reason and a "Try again" button. The existing manual "Push to Drive" / "Pull from Drive" buttons are kept but moved behind a small "Manual..." disclosure inside the popover, so the default surface is auto-sync but a power user can still force a direction. Implementation notes: the debounce timer lives in a new `driveAutoSync.js` module — `scheduleAutoSync()` resets a `setTimeout(performAutoSync, 10000)`, called from the same `driveSyncStateChanged` CustomEvent the indicator already listens to. `performAutoSync` is a single async function holding the decision-tree branch logic; both auto-push and auto-pull paths call back into the existing `exportTodosToDrive` / `importTodosFromDrive` orchestrators (don't fork them — the rename PR established `writeLastDriveSyncedAt` as the single sync-marker write point, and auto-sync must go through the same path so the indicator stays correct). `importTodosFromDrive` gains an `{ silent: true }` opts hint that skips the confirmation modal for the auto-pull path; manual pulls keep the modal. Define a module-level `_autoSyncArmed` flag (per-session boolean) that gates whether triggers actually fire — flipped to true by the first successful manual click of the session, flipped to false by a `failed` or `diverged` state. Important data-loss guard: BEFORE calling `exportTodosToDrive` for an auto-push, re-check that `driveAhead` is still false by re-querying Drive (the cached value might be stale — another device could have pushed during the 10s debounce window). If the re-check shows Drive moved, abort the push and re-enter the decision tree (probably going to `diverged`). Without this re-check, auto-push has a race that silently overwrites a pushed-from-another-device version of the file. Out of scope for this slice: window-focus trigger, periodic poll, `beforeunload` "flush pending push", multi-file Drive history, conflict-merge UI. Each of those is a separate future entry once we see real usage. Acceptance criteria: (1) after the first manual click of a session, editing todos triggers a Drive push within ~10s of the last edit, without user intervention; (2) opening the app when Drive is newer (another device pushed) auto-pulls on load without prompting, indicator briefly shows pull-in-progress then green; (3) editing locally while Drive is simultaneously newer (diverged) leaves auto-sync disarmed and surfaces the red `ti-cloud-x` with a popover offering explicit overwrite-with-warning buttons; (4) network failure or token expiry sets indicator to red `ti-cloud-off`, disarms auto-sync, surfaces failure reason in popover with a "Try again" button; (5) the pre-push driveAhead re-check prevents the silent-overwrite race when another device pushes during the debounce window. Tests: in a new `driveAutoSync.test.js`, pin the four-branch decision logic against the timestamp matrix (16 combinations of three booleans plus the armed/disarmed gate); pin the debounce — N rapid mutations within 10s result in exactly one `performAutoSync` call; pin the diverged auto-pause behavior; pin the pre-push driveAhead re-check via a stubbed `queryLatestDriveFile` that returns a newer modifiedTime between the schedule and the fire; pin that auto-pull passes `{ silent: true }` to `importTodosFromDrive` and a manual pull does not. In `driveImport.test.js`, pin that `{ silent: true }` opts skip the confirmation modal but otherwise run the full import pipeline (including the `fromSync: true` thread). In `driveExport.test.js`, pin no behavior change for manual export and the new pre-export re-check for auto-export. Changelog: add a 1.1 entry "New: Drive auto-sync — after your first manual sign-in of a session, the app pushes and pulls automatically as you edit. Conflicts pause auto-sync and ask you to resolve manually."
