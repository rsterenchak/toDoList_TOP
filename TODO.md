- [x] **[HIGH]** Fix mobile title vanishing on first tap of read-mode row
  - Description: On mobile (≤420px), tapping a committed row to enter read mode collapses the title slot to an empty band — neither the wrappable `.toDoTitleDisplay` span nor `#toDoInput` is visible, even though the row-level active styling (indigo wash, left-edge border) and the description panel below both render correctly. Root cause is the `:focus-within` rule introduced for the title display→input swap: `#toDoChild:not([data-original-blank="true"]):focus-within .toDoTitleDisplay { display: none; }`. The first-tap branch of `wireToDoRowClick` calls `descToggle.click()` to open the description panel, which on Android Chrome and most non-iOS engines focuses the `<button>` it synthetically clicks. That focus on a descendant of `#toDoChild` makes `:focus-within` match, which hides the span. Meanwhile `#toDoInput` itself only un-hides on its own `:focus` (not `:focus-within`), so the input stays `opacity: 0; position: absolute` and invisible. The result: read mode opens, the wash and border paint, the description appears, but the title row is empty until a second tap focuses the input directly. Replace `:focus-within` with an explicit attribute the click handler sets only when entering true edit mode, so read mode keeps the span visible and edit mode hands off to the input.
    - Behavior:
      1. First tap on a collapsed committed row (read mode) → `data-mobile-read="true"` set on `#toDoChild`, `.toDoTitleDisplay` stays visible and unclamps to wrap, `#toDoInput` stays hidden. No `:focus-within` dependency.
      2. Second tap on the active row's title area → `data-mobile-edit="true"` set on `#toDoChild` (new attribute), CSS swaps: span hides, input shows, `wireToDoRowClick` calls `toDoInput.focus()` with caret at end.
      3. Blur on `#toDoInput` → handler clears `data-mobile-edit` from `#toDoChild`, CSS swaps back: input hides, span reappears with the now-current `item.tit`.
      4. The existing `data-mobile-read` clearing path on `descToggle` close also clears `data-mobile-edit` defensively, so closing the description always collapses the row fully back to single-line.
    - Implementation notes:
      - Remove the `#toDoChild:not([data-original-blank="true"]):focus-within .toDoTitleDisplay { display: none; }` rule entirely. It's the source of the bug and is now redundant.
      - Remove the `#toDoChild:not([data-original-blank="true"]) #toDoInput:focus { ... }` rule that un-hid the input on its own focus. Replace both with attribute-keyed rules on `#toDoChild[data-mobile-edit="true"]`:
        - `#toDoChild[data-mobile-edit="true"]:not([data-original-blank="true"]) .toDoTitleDisplay { display: none; }`
        - `#toDoChild[data-mobile-edit="true"]:not([data-original-blank="true"]) #toDoInput { opacity: 1; position: static; width: auto; height: auto; pointer-events: auto; }`
      - In `wireToDoRowClick`'s mobile branch, after the first-tap `data-mobile-read` block returns, the second-tap path (the existing committed-row activation that calls `toDoInput.focus()`) must also set `data-mobile-edit="true"` on `#toDoChild` before the focus call. The order matters — set the attribute first so the CSS swap has happened by the time focus lands, otherwise the input is still `opacity: 0; pointer-events: none` and the focus call is a no-op.
      - Add a blur listener on `#toDoInput` in `buildToDoRow` (alongside the existing snap-back blur handler) that clears `data-mobile-edit` from `toDoChild`. The existing blur handler already runs `toDoInput.title = item.tit || ""` and `toDoTitleDisplay.textContent = item.tit || ""` — extend it with `toDoChild.removeAttribute('data-mobile-edit')`.
      - In the `descToggle` click listener that already clears `data-mobile-read` on close, also clear `data-mobile-edit` for safety — closing the description should fully collapse the row regardless of what state the title was in.
      - The `#toDoChild:not([data-original-blank="true"]) #toDoInput` absolute-positioning block stays — that's what keeps the hidden input out of the layout flow. Only the `:focus` un-hide rule changes (now attribute-keyed instead of pseudo-class-keyed).
    - Acceptance criteria:
      - Tapping a collapsed committed row on mobile (≤420px) opens read mode with the title wrapping into multi-line text, the indigo wash and left-edge border applied, and the description panel open below — no empty title band.
      - Tapping the title area of an already-active row swaps to the input with caret at end, soft keyboard appears, the user can edit normally.
      - Tapping outside or blurring the input restores the span with the updated title, no flash of empty space.
      - Closing the description panel (via the chevron or via tapping another row) collapses the row fully — title returns to single-line truncation, both `data-mobile-read` and `data-mobile-edit` cleared.
      - No regression on desktop or tablet (>420px) — the new `data-mobile-edit` rules sit inside the `@media (max-width: 420px)` block alongside the existing mobile chrome.
    - Out of scope: the original read-mode visual treatment (indigo wash, left-edge border, description auto-open) — those are working as intended. The row-level click handler's bail-outs for the copy button, due pill, and checkbox are also untouched.
