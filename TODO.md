- [x] **[MEDIUM]** Surface a generated draft that has not been looked at — Completed: 2026-07-23
  - Type: feature
  - Description: Generate dispatches to Actions and takes roughly a minute, so the user closes the task and does something else. When the derived entry lands in the description, nothing announces it — the task renders identically to an idea that was never touched, and the draft is found only by reopening the task on a hunch. REVIEW and ASKING both reach the user through the badge word on mobile rows; a landed draft is the third state genuinely blocked on a human and is the only one still silent. Give it the same treatment: a `⌁ DRAFTED` badge state, cleared once the task has been opened. The `draft_seen_at timestamptz` column already exists on `todos` — treat it as present and wire to it exactly as `entry_reviewed_at` is wired; do not gate on verifying it, since the repo cannot see the live schema and a previous run stalled indefinitely trying.
  - Behavior: A task whose linked `agent_queue` row is in `drafted` and whose todo has no `draftSeenAt` renders `⌁ DRAFTED` in amber, on mobile as well as desktop, joining REVIEW and ASKING as a blocked-on-you state. Opening the task's description editor clears it — the signal means "you have not looked at this yet", and looking is the whole of what it asks for, so no separate dismiss control. Tapping the badge opens the description editor, matching what ASKING already does. The state is per-task and does not clear for other generated tasks. Once cleared it does not return for that generation; a subsequent Generate on the same task sets it again.

- [x] **[MEDIUM]** Project switcher: show which other projects have a triage question waiting — Completed: 2026-07-23
  - Type: feature
  - Description: A triage question parked in `needs_words` renders as an `⌁ ASKING` badge on a task row, which means it is only visible inside the project currently on screen. With seven projects, a question waiting in another one is silent until the user happens to switch to it — and the point of dispatching work asynchronously is that the user leaves. Show a per-project count of waiting questions in the project switcher. Deliberately narrow: this counts ONLY `agent_queue` rows in `needs_words`, which requires no todo data at all. A previous attempt also counted landed drafts, which needed each todo's `draftSeenAt` for projects whose todos are not loaded, and that broke the switcher's render entirely — see Out of scope.
  - Behavior: Each project in the switcher shows an amber count of its `agent_queue` rows in `needs_words`. A project with none shows nothing — no zero badge. The open project shows its own count the same way, so the switcher reads uniformly rather than special-casing the current one. The count is live: a realtime `agent_queue` push updates it without a re-render or a project switch. It is labelled as questions specifically, not as "needs you", since shipped-but-unreviewed entries and landed drafts are not included. Tapping a project switches to it exactly as it does today; the count adds no tap behavior. Above all, the switcher renders its project list normally whether or not any count can be resolved.
  - Implementation notes:
    - HARD REQUIREMENT — the count must not be able to break the switcher. Compute all counts into a plain map BEFORE the render loop, inside a try/catch that falls back to an empty map, and inside the loop read the map with a default of zero. Never call into the store, `listLogic`, or any lookup from inside the per-project render body. The previous attempt threw inside that loop and aborted the whole render, leaving the app with tab chrome and no project header or task list — and every revert control lives inside the app, so the failure is not recoverable in-app.
    - This entry needs NO todo data. Do not call `listItems`, do not read `draftSeenAt`, do not resolve todos for any project. The count is derived entirely from `agent_queue` rows and their `project_id`.
    - `agentQueueStore.js` owns the fetch. Add an all-projects load beside the per-project one: `fetchQueueRows` filters on `project_id`, but RLS already scopes `agent_queue` to the user, so the same select with no `.eq` returns every project's rows in one round trip. Cache it SEPARATELY from `_rows` — that cache is the render path's synchronous source for `getQueueRowForTodo`, and re-scoping it would change what every task row derives.
    - Follow `fetchQueueRows`'s existing shape exactly: wrap the chain in try/catch and resolve to `[]` on error, since the test client's `.select()` resolves immediately and has no `.eq`. A failed load degrades to no counts, never a throw.
    - Counts key off `project_id`, but the switcher renders project NAMES. Resolve name → id through the same `listLogic.getProjectId` the store already uses in `loadQueueRows`, and treat an unresolvable name as zero rather than an error — a project row can exist in the DOM before its id is known.
    - Reuse the store's existing realtime channel and `onQueueChange` rather than opening a second subscription; the channel is already app-lifetime and fires on every `agent_queue` change regardless of project. Refresh the all-projects cache on that same push.
    - Do not import `agentView.js` or `toDoRow.js` into the switcher — read the store directly, as the row layer does.
    - Tests must cover the failure mode, because the stub Supabase client resolves everything to `[]` and never exercises the real path: add a regression test that the switcher renders its full project list when the count source throws, and one that a project with no resolvable id renders without a badge.
    - `style.css`: reuses the existing `#ffbd5e` amber and the established radius conventions; no new tokens. This entry touches `style.css`, so it runs with no other style-touching entry in flight.
  - Out of scope: Counting landed drafts (`drafted` with no `draftSeenAt`) cross-project — it requires other projects' todos in memory, which is not guaranteed, and is what broke the previous attempt. Counting REVIEW cross-project, which would mean a per-repo TODO.md marker read for all seven projects on load. Any notification outside the app, a global aggregate badge on the chat button, auto-switching to a project with waiting work, the task rows' own badges, and the Agent board.
  - File: `toDoList_main/src/agentQueueStore.js`, `toDoList_main/src/main.js`, `toDoList_main/src/style.css`
