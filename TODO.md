## Bugs

- [ ] **[MEDIUM]** Fix recurring-task flash animation cancelling itself by re-parenting the row mid-animation
  - Description: When the user checks a recurring todo, the row briefly flashes (the `recurring-flash` class added in `wireCheckbox`) for ~250ms before resetting the checkbox and advancing the due date — but the flash often doesn't visibly play because the same change handler then calls `listLogic.sortCompletedToBottom(projectName)` + `reorderToDoDOM(projectName)` synchronously on the same tick. `reorderToDoDOM` calls `mainDiv.appendChild(row)` on the live row to move it into its new slot (the recurring branch spawns a completed clone, so the row order changes); the browser treats `appendChild` on an already-mounted element as a DOM move and cancels any running CSS animation on that element. The class is still applied, but the keyframes are reset to frame 0 in the new DOM position and the user sees little to nothing. This is the same root cause as the slide-out fade completion animation bug — same pattern, same fix, different class. Fix by deferring the `sortCompletedToBottom` + `reorderToDoDOM` calls inside the recurring branch until the flash's `setTimeout` fires (or, preferably, until an `animationend` listener resolves — but the existing `recurring-flash` is currently driven by a 250ms `setTimeout` rather than CSS keyframes, so a `setTimeout`-based defer is the minimum change). Concretely: move the `sortCompletedToBottom` + `reorderToDoDOM` calls from their current position immediately after `applyDueUrgency` into the `setTimeout` callback that runs after the flash, AFTER the `recurring-flash` class is removed and `checkToDo.checked = false` lands. Under `prefersReducedMotion()` the reorder runs synchronously as today, since no animation needs protection. Persistence is unaffected — `advanceRecurringTodo` already mutates the data model on the same tick the user clicked; only the DOM-side reorder is deferred. Test pins: clicking a recurring todo's checkbox adds `.recurring-flash` and only calls `reorderToDoDOM` after the flash's `setTimeout` resolves; under `prefersReducedMotion()` the reorder runs synchronously; the data-model state (the completed clone added by `advanceRecurringTodo`) is present on the same tick as the click regardless of animation timing.

- [x] **[MEDIUM]** Add selectable ambient visualizer to focus music popover for when YouTube video is blocked
  - Description: When YouTube video is blocked on corporate networks the audio still plays but the 240×135 iframe shows nothing, leaving the popover visually inert. Add a generative ambient visualizer that occupies the iframe's footprint while music is playing, toggled by a new checkbox in the popover and styled by a dropdown next to it. Important constraint: the visualizer CANNOT react to the YouTube audio waveform — YouTube iframes don't expose their audio stream to Web Audio API across origins, so any analyser-driven design is impossible without abandoning the YouTube embed. The five visualizer styles are decorative loops on fixed timings, not audio-reactive: STARFIELD (slow horizontal star drift over a deep purple void, most "Void theme"), BLOBS (three slow-drifting blurred lavender/purple blobs on long sine paths, lava-lamp feel), RINGS (concentric purple rings expanding from a central core on a 4-second cycle), BARS (16 staggered equalizer-style bars with offset CSS keyframes — looks the most "music visualizer" while being decorative), GHOST (companion ghost mascot bobs gently with music notes rising past, ties to the existing ghost SVG). Picker layout follows PICK-A from the mockup round: a single row below the station list containing the checkbox (left), a STYLE label, and a native `<select>` dropdown listing the five styles by name. New row's vertical position sits between the curated stations list and the Pause + slider controls row so the popover keeps its existing top-down rhythm. New module `musicVisualizer.js` mirrors `music.js`'s singleton pattern: exports `ensureVisualizer()` (mount into the iframe-footprint container), `destroyVisualizer()` (tear down, stop all animations), `setVisualizerStyle(styleId)` (swap the active style without remounting the container). The iframe and visualizer share the same wrapper element with absolute-positioned children — visualizer on top with a higher z-index when enabled, iframe behind. When the user unchecks the box, the visualizer detaches and the YouTube iframe is revealed again (useful at home where video isn't blocked). Critical to honor the prior visualizer-failure root cause noted in past notes: any CSS-only bar visualizer (BARS option) MUST set an explicit parent height on the visualizer wrapper because flex-end alignment of percentage-height bars collapses to zero against an unsized parent. Set `height: 100%` on the wrapper and let aspect-ratio: 16/9 on the iframe container drive the actual pixel height. Persistence: two new preferences in `prefs.js` — `todoapp_musicVisualizerEnabled` (boolean, defaults to false so first-time users keep current behavior) and `todoapp_musicVisualizerStyle` (string, one of "starfield" / "blobs" / "rings" / "bars" / "ghost", defaults to "starfield"). Visualizer only runs when music is actively playing — when the user pauses, animations pause too (set `animation-play-state: paused` on the active style's root element, no need to tear down). When the user resumes, animations resume from where they left off. Visualizer also bails on `prefers-reduced-motion: reduce` even when the checkbox is on — substitute a static frame (single rendered snapshot of the chosen style, no animation) so the popover doesn't look broken. Tests: each style mounts a root element with the expected class (e.g., `.musicViz--starfield`), the checkbox writes the pref and toggles visualizer presence, the dropdown writes the style pref and swaps the active style without unmounting the wrapper, prefers-reduced-motion swaps the animated style for a static fallback.

  - Description: Clicking the row checkbox to mark a todo complete currently produces no completion animation — the row's "done" styling (strikethrough title, muted color) applies instantly, which reads as no feedback at all and makes the interaction feel flat.   - Description: Add a new "Export to Drive" menu item directly below the existing "Export JSON" row in the ghost popover menu. On success, show a toast matching the existing toast styling with an "Open in Drive" link to the uploaded file's `webViewLink`. Re-prompt OAuth on each session — do not persist the access token to `localStorage`. Reuse the existing export filename pattern (`todoapp-export-YYYY-MM-DD.json`) and the same JSON serialization path so both export methods produce byte-identical files.
  - Behavior:
    1. User opens ghost popover menu, sees "Export to Drive" as a new row below "Export JSON".
    2. First click in a session opens Google sign-in popup; subsequent clicks in the same session reuse the in-memory token until it expires (~1 hour), at which point re-auth runs silently if possible, otherwise re-prompts.
    3. While the upload is in flight, the menu row shows a subtle loading state (text dim + spinner) and is disabled to prevent double-uploads.
    4. On success: toast "Exported to Drive" with "Open in Drive" link (opens `webViewLink` in a new tab).
    5. On failure (denied consent, network error, API error): toast with a clear error message; no partial state left behind.

    - Use the `drive.file` scope only (not `drive`) so the app stays in the non-sensitive scope tier and avoids Google's security review process.
    - Extract the JSON serialization logic from the existing Export JSON handler into a shared helper so both code paths produce identical output.
    - Build the OAuth client ID into a module-level constant; do not hardcode in `main.js`.
  - Out of scope: folder picker (always dumps to Drive root), token persistence across sessions, two-way sync, import-from-Drive (separate future entry), accounts/multi-user.
