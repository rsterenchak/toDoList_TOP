  - Description: The Drive sync indicator (green `ti-cloud-check` / amber `ti-cloud-up` overlay on the ghost icon + DRIVE section header) stays amber after a successful Import from Drive, falsely telling the user they're still behind. Fix by renaming the marker to `lastDriveSyncedAt` to reflect that it tracks the last sync in *either* direction, and writing it from both Drive flows: `driveExport.js` continues writing `now.toISOString()` inside its upload-success then-handler (unchanged semantics, new key name), and `driveImport.js` writes the Drive file's `modifiedTime` (from the `file` descriptor returned by `queryLatestDriveFile`) inside its post-`importTodosFromString` success branch — using `modifiedTime` instead of `Date.now()` guarantees local-time equals Drive-time after import, so the sync-state comparison reads as `synced` regardless of clock skew. Write unconditionally on import success (don't gate on "data actually changed") — after a user-initiated import the local state is by definition in sync with that Drive file, and a noop-import that still leaves the indicator amber would be confusing. In `prefs.js`, rename the storage key constant and its read/write helpers (`LAST_DRIVE_SYNCED_AT_KEY`, `readLastDriveSyncedAt`, `writeLastDriveSyncedAt`); the underlying localStorage key string also moves from `todoapp_lastDriveExportedAt` to `todoapp_lastDriveSyncedAt`. Add a one-shot migration on app load that reads any legacy `todoapp_lastDriveExportedAt` value, writes it under the new key if the new key is empty, and removes the legacy key so existing users don't lose their "last exported" state. Update all call sites: `driveExport.js` imports `writeLastDriveSyncedAt` instead of `writeLastDriveExportedAt`; `driveImport.js` adds the same import and the new write; `main.js` swaps `readLastDriveExportedAt` for `readLastDriveSyncedAt` everywhere it feeds the relative-time label and the sync-state computation.
